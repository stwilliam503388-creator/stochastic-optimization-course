# 附录B：TOP 5 必踩坑

> 这些坑我本人（和很多前辈）都踩过。提前知道，至少省你一周的 debug 时间。

---

## 坑 #1：场景太少 → 低估风险

**症状**：用 3-5 个场景做了随机规划，结果在实际测试时表现远差于预期。

**为什么**：3-5 个场景不足以代表连续分布。场景太少时，模型学会了「过拟合」这 3-5 个特定场景，而不是学到分布的整体特征。

**生活类比**：你只看天气预报的「晴/阴/雨」三种情况决定明天穿什么——结果第二天下了冰雹。

**具体数字**：

```
场景数 N     | 估计误差（相对）
------------|---------------
5           | ~30-40%
50          | ~10-15%
500         | ~3-5%
5000        | ~1-2%
```

**怎么避免**：
- **最少 50 个场景**起步（除非问题有解析解——比如新闻摊贩）
- 用「out-of-sample 测试」验证场景代表性：用已生成的场景训练模型，再用另一组独立生成的场景评估表现。如果表现大幅下降 → 场景太少

**✅ 正确做法**：
```
1. 生成 N=200 场景作为训练集
2. 生成另一个 N=200 场景作为测试集
3. 在训练集上求解
4. 在测试集上评估——如果结果接近 → N 够了
```

## 坑 #2：分布假设错误 → 方向性偏误

**症状**：假设需求服从正态分布，实际是厚尾（fat-tailed）分布。模型在大多数时候表现正常，但每隔几个月来一次「极端事件」模型就崩了。

**为什么**：正态分布假设低估了尾部概率。正态分布下，三倍标准差之外的事件概率约 0.3%——但在真实业务数据中，「极端事件」的概率可能高达 5-10%。

**生活类比**：你假设身高服从正态分布准备了一扇 2.2 米高的门——99.9% 的人都能通过。但你以为 NBA 球员来的概率是 0.1%，实际可能是 10%。

**具体例子**：假设需求均值为 100，标准差为 20。正态分布下，需求 > 160 的概率约 0.13%。但在真实（厚尾）分布下，需求 > 160 的概率可能是 5%。你用正态假设做的库存方案，每 20 天就会缺一次货而不是 770 天。

**怎么避免**：
- **先做分布检验**：用 Kolmogorov-Smirnov 检验或 Q-Q 图检查数据是否符合正态假设
- **不放心就用非参数方法**：SAA（样本均值近似）不假设分布形态——直接用历史数据作为场景
- **或者直接用鲁棒优化**：不需要任何分布假设

## 坑 #3：鲁棒优化过度保守 → 方案不可用

**症状**：你用了箱式不确定性集合（Box Uncertainty Set），要求所有参数同时取最坏值。结果方案保守到「多建了 2 倍的产能，但利用率只有 30%」，业务部门直接把你的方案扔进垃圾桶。

**为什么**：箱式不确定性集合假设所有参数同时取最坏值——这在真实世界几乎从不发生。10 个需求参数同时处于最坏边界，概率几乎是零。

**生活类比**：你为「明天全城所有红绿灯同时坏掉」这种情况规划通勤路线——你会准备一个极其绕路的方案，但明天所有红绿灯正常工作时这个方案看起来就很蠢。

**怎么避免**：

```
保守度递减：
箱式（Box）         → 最保守 ❌
   ↓
椭球式（Ellipsoidal）→ 中等
   ↓
预算式（Budget）    → 可控 ✅   ← 工业界推荐
   ↓
Γ = 0              → 无保守（= 确定性模型）
```

**✅ 正确做法**：
1. 从预算不确定性（Budget Uncertainty）开始
2. 设 Γ = N/3（最多 1/3 的参数可同时偏离）
3. 逐步增大 Γ 直到结果「在可接受的安全范围内」
4. 和业务团队讨论：「多花 10% 成本来防御 X% 的极端情况，你们接受吗？」

## 坑 #4：忘记非预期约束 → 模型「作弊」

**症状**：两阶段随机规划的结果好得出奇——甚至比 true optimal 还好。这不是你厉害，是模型在作弊。

**为什么**：非预期约束（Non-anticipativity）要求第一阶段决策不能依赖于第二阶段信息。如果你忘了加上这个约束，模型会让第一阶段决策「知道未来」，从而得到不现实的最优解。

**生活类比**：你和一个朋友玩猜拳——如果你能在出拳之前看到对方的手，那你永远能赢。但这违反了「同时出拳」的规则。

**数学上**：非预期约束就是要求第一阶段变量 x 对所有场景相同：

```
✅ 正确：x₁ = x₂ = ... = x_N  （所有场景共享同一个第一阶段决策）
❌ 错误：x₁ ≠ x₂               （每个场景自己做第一阶段决策——这是作弊）
```

**怎么避免**：建模时检查：第一阶段变量有没有场景下标？如果有，就是忘了非预期约束。

**自检清单**：
```
□ 第一阶段变量没有场景下标 s
□ 第二阶段变量有场景下标 s
□ 第二阶段约束的上界绑定到了 s
□ 目标函数中 p_s * f(x, y_s) 正确求和
```

## 坑 #5：忽略计算代价 → 模型跑不出结果

**症状**：信心满满地建了一个 10,000 场景的随机规划对等模型，结果求解器跑了 12 小时还没出结果。

**为什么**：对等模型的规模 = 原问题规模 × 场景数。你以为「加个循环就行」，但求解器不是在解同一个问题 10,000 次——它在解一个大了 10,000 倍的问题。

**具体数字**：

| 原问题 | 场景数 | 对等模型规模 | 求解时间（估计） |
|-------|-------|------------|---------------|
| 100 var, 50 con | 10 | 1,100 var, 550 con | 1 秒 |
| 100 var, 50 con | 1,000 | 100,100 var, 50,050 con | 10 秒 |
| 100 var, 50 con | 100,000 | 10M var, 5M con | 可能需要分解 |
| 1,000 var, 500 con | 10,000 | 10M var, 5M con | Benders 或放弃 |

**怎么避免**：
1. **先小规模验证逻辑**：用 5-10 个场景跑通模型，确保公式和代码正确
2. **预估最终规模**：按最大场景数估算对等模型的大小
3. **选择合适的方法**：
   - 场景 < 10,000 → 直接求解对等模型
   - 场景 10,000 ~ 100,000 → Benders 分解 / L-shaped
   - 场景 > 100,000 → 渐进式对冲 / 情景缩减
4. **场景缩减**：用聚类算法把 10,000 场景缩减到 100 个代表性场景

> 🆘 逃生通道：如果你已经踩了坑，先停下来。回到 [02-methods.md](02-methods.md) 的 2.3 节重新理解对等模型的规模问题，然后根据你的问题规模选择合适的方法。
